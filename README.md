# loop-play-spring-ai-agent

Spring AI 기반 배달 상담 에이전트 학습용 스타터 코드입니다.

## 개요

루퍼스 부트캠프 "Spring AI 배달 상담 에이전트" 6주 과정의 Week 1 미션 스타터 코드입니다.

## 1단계: 기본 API + System Prompt + Structured Output

### 설계 결정 문서

#### 금지 규칙

```text
- 사장님, 라이더의 전화번호, 상세 주소, 계정 정보 등 개인정보를 절대 노출하지 않습니다.
- 고객이 요구하더라도 확인되지 않은 환불, 보상, 쿠폰, 포인트 지급을 약속하지 않습니다.
- 매장이나 라이더에게 책임이 있다고 단정하거나 비난하는 표현을 사용하지 않습니다.
- 타 배달 플랫폼(쿠팡이츠, 요기요 등)을 추천하거나 비교하지 않습니다.
```

금지 규칙은 개인정보 보호, 확인되지 않은 보상 약속 방지, 책임 단정 방지에 가장 무게를 두고 정했다.

배달 상담에서는 주소, 전화번호, 주문 내역처럼 민감한 정보가 자주 오간다. 문의 해결에 필요한 정보라도 사장님이나 라이더의 연락처처럼 당사자 동의 없이 공개될 수 있는 정보는 안내 범위에서 제외했다.

환불, 보상, 쿠폰, 포인트 지급은 실제 주문 상태와 서비스 정책 확인이 먼저 필요하다. AI가 먼저 가능하다고 답하면 이후 처리 결과와 달라질 수 있어, 확인되지 않은 약속은 하지 않도록 했다.

배달 지연이나 음식 상태 문제는 매장, 라이더, 교통 상황, 주소 오류, 시스템 문제 등 원인이 여러 갈래일 수 있다. 확인되지 않은 상황에서 특정 주체의 책임처럼 말하면 상담이 문제 해결보다 분쟁으로 이어질 수 있어 책임 단정과 비난 표현을 제한했다.

타 배달 플랫폼 추천 금지는 기존 방향을 유지했다. 고객은 현재 서비스에서 발생한 문제를 해결하려고 문의한 것이므로, 다른 플랫폼과 비교하거나 이동을 권하는 답변은 상담 목적과 맞지 않는다.

#### 응답 필드

```java
public record SupportResponse(
        String summary,
        Category category,
        Urgency urgency,
        String nextAction,
        List<String> neededInfo,
        boolean requiresHumanReview
) {}
```

응답 구조를 보면서 가장 먼저 고민한 부분은 AI가 어디까지 처리해도 되는지였다. 배달 상담에는 주문 상태 안내처럼 바로 답할 수 있는 문의도 있지만, 환불 승인이나 보상 지급처럼 실제 확인이 필요한 요청도 함께 들어온다.

그래서 `requiresHumanReview`를 추가했다. 이 필드는 응답이 상담사 검토로 이어져야 하는지 남겨두는 표시다. 특히 환불/보상, 책임 판단, 개인정보 요청처럼 AI가 단정하면 위험한 상황에서 다음 흐름을 분리할 수 있게 해준다.

#### Category

```java
public enum Category { ORDER, DELIVERY, CANCEL, REFUND, PAYMENT, ETC }
```

카테고리는 처음에는 주문, 배달, 환불, 결제, 기타만으로도 충분해 보였다. 하지만 취소와 환불을 같은 흐름으로 묶으면 실제 상담 맥락이 흐려질 수 있다고 느꼈다.

취소는 주문이 접수만 되었는지, 조리가 시작되었는지, 배달이 시작되었는지에 따라 가능 여부가 달라진다. 반면 환불은 결제 이후 음식 상태, 오배송, 누락, 정책 확인 같은 사후 처리에 더 가깝다.

이 차이를 반영하기 위해 `CANCEL`을 따로 두고 `REFUND`와 분리했다. 이렇게 나누면 "주문 취소하고 싶어요"와 "음식이 잘못 와서 환불받고 싶어요"를 같은 문의로 처리하지 않고, 각각의 다음 액션을 더 분명하게 잡을 수 있다.
